# 分析技术报告

## 1. 报告目标

本报告说明首版分析系统如何从视频、评论、回复这些公开内容中提取：

1. 事件脉络
2. 情感分布
3. 观点/立场
4. 高活跃传播线索

## 2. 首版分析对象

本项目选取“线下突发事件在网络上的反应”作为主要场景，用一个“地铁晚高峰故障”样例数据集演示方法链路。

原因有三点：

1. 该场景同时具备时效性、情绪性和传播性。
2. 评论区通常能快速暴露公众真实关注点。
3. 容易观察到从“现场记录”到“责任追问”的话题迁移。

## 3. 分析流程

### 3.1 数据输入

输入为统一 JSON 数据结构，包含：

1. 视频元数据
2. 评论列表
3. 回复列表

### 3.2 预处理

1. 将时间统一转为 ISO 格式。
2. 合并视频文本、评论文本和回复文本。
3. 清洗标点与空字符。
4. 保留用户与视频之间的关联关系。

### 3.3 分析模块

#### 模块 A：情感识别

首版采用规则法作为基线：

1. 构建正向、负向、中性提示词。
2. 对评论和回复分别打分。
3. 汇总到视频级和事件级。

优点：

1. 实现简单
2. 易解释
3. 适合作为第一版验证

局限：

1. 反讽识别弱
2. 上下文依赖弱
3. 对新表达泛化有限

#### 模块 B：观点/立场提取

首版采用主题词和规则映射识别以下立场：

1. `support_response`
   支持现场处置、强调先恢复秩序。
2. `criticize_response`
   质疑响应速度、信息透明度、责任归属。
3. `call_for_amplification`
   号召转发、曝光、冲热搜、持续监督。
4. `rumor_warning`
   提醒辟谣、指出带节奏、反对夸大叙事。

#### 模块 C：事件脉络分析

基于视频发布时间和评论时间，观察：

1. 现场曝光阶段
2. 官方回应阶段
3. 恢复与辟谣阶段
4. 追责与反思阶段

#### 模块 D：高活跃传播线索识别

这里只做“线索识别”，不做定性断言。

判断依据：

1. 同一用户是否跨多个相关视频发声
2. 是否频繁使用“转发/曝光/热搜”等放大词
3. 文本是否高度重复

## 4. 输出结果

脚本 `scripts/analyze_public_douyin.py` 会输出两类文件：

1. `outputs/analysis_report.md`
   面向阅读汇报。
2. `outputs/analysis_summary.json`
   面向后续程序接入。

## 5. 结果解读框架

首版建议重点关注以下指标：

1. 总视频数、评论数、回复数
2. 负向情绪占比
3. 主要观点簇
4. 哪个视频下争议最强
5. 哪些用户表现出高活跃放大线索

## 6. 结果意义

如果首版结果能较稳定地输出这些指标，就意味着项目已经具备三个基础能力：

1. 能把抖音公开内容转成研究数据
2. 能对事件舆论进行结构化分析
3. 能为后续预警与研判提供第一层信息

## 7. 首版技术判断

首版不追求模型复杂度，优先追求：

1. 流程完整
2. 结论可解释
3. 输出可汇报

因此，规则法是合理的起点；当真实样本积累后，再升级到更强模型。
